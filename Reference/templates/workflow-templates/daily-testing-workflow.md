# 日常测试工作流

## 工作流概述

**适用场景**: 日常开发过程中的持续测试活动  
**涉及 Skills**: functional-testing, automation-testing, bug-reporting  
**执行频率**: 每日

---

## 工作流步骤

### 早晨启动（9:00-9:30）

#### 1. 环境检查
```markdown
- [ ] 测试环境状态正常
- [ ] CI/CD 流程运行正常
- [ ] 自动化测试通过率 > 95%
- [ ] 无阻塞性缺陷
```

#### 2. 任务规划
```markdown
- [ ] 查看今日测试任务
- [ ] 确认优先级
- [ ] 评估工作量
- [ ] 识别依赖和风险
```

**输出**: 今日测试计划

---

### 上午测试（9:30-12:00）

#### 3. 新功能测试
**使用 Skill**: `functional-testing`

**活动**:
```markdown
1. 获取最新代码/构建
2. 执行冒烟测试
3. 执行功能测试用例
4. 记录测试结果
```

**检查清单**:
- [ ] 核心功能正常
- [ ] 新功能符合需求
- [ ] 无明显缺陷
- [ ] 性能可接受

#### 4. 缺陷验证
**使用 Skill**: `bug-reporting`

**活动**:
```markdown
1. 验证已修复的缺陷
2. 执行回归测试
3. 更新缺陷状态
4. 记录验证结果
```

**缺陷状态流转**:
```
待修复 → 已修复 → 验证中 → 已关闭
                  ↓
                重新打开（验证失败）
```

---

### 午间同步（12:00-13:00）

#### 5. 进度同步
```markdown
- [ ] 更新测试进度
- [ ] 同步阻塞问题
- [ ] 与开发团队沟通
- [ ] 调整下午计划
```

**同步内容**:
- 测试完成情况
- 发现的问题
- 需要的支持
- 风险和建议

---

### 下午测试（13:00-17:00）

#### 6. 深度测试
**活动**:
```markdown
1. 执行复杂场景测试
2. 执行集成测试
3. 执行探索性测试
4. 性能和安全测试
```

**测试重点**:
- 边界条件
- 异常场景
- 数据一致性
- 用户体验

#### 7. 自动化维护
**使用 Skill**: `automation-testing`

**活动**:
```markdown
1. 修复失败的自动化用例
2. 添加新的自动化用例
3. 优化测试脚本
4. 更新测试数据
```

**自动化健康度指标**:
- 通过率: > 95%
- 执行时间: < 30 分钟
- Flaky 测试: < 5%

---

### 晚间总结（17:00-18:00）

#### 8. 日报编写
```markdown
- [ ] 总结今日测试工作
- [ ] 记录发现的问题
- [ ] 更新测试进度
- [ ] 规划明日工作
```

**日报模板**:
```markdown
# 测试日报 - [日期]

## 测试概况
- 执行用例: [数量]
- 通过用例: [数量]
- 失败用例: [数量]
- 新增缺陷: [数量]

## 主要工作
1. [工作项 1]
2. [工作项 2]
3. [工作项 3]

## 发现的问题
1. [问题 1] - [严重程度]
2. [问题 2] - [严重程度]

## 阻塞问题
- [阻塞问题描述]

## 明日计划
1. [计划 1]
2. [计划 2]
```

#### 9. 环境清理
```markdown
- [ ] 清理测试数据
- [ ] 释放测试资源
- [ ] 备份重要数据
- [ ] 准备明日环境
```

## 工作流健康度评分

### 评分维度

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 测试覆盖率 | 25% | 计划用例执行率 |
| 缺陷发现率 | 20% | 新发现缺陷数量 |
| 自动化稳定性 | 20% | 自动化测试通过率 |
| 响应速度 | 15% | 缺陷验证及时性 |
| 文档完整性 | 10% | 测试记录完整度 |
| 团队协作 | 10% | 沟通和同步效率 |

### 健康度等级

- **优秀** (90-100分): 流程顺畅，质量高
- **良好** (80-89分): 基本达标，有改进空间
- **一般** (70-79分): 存在问题，需要优化
- **较差** (<70分): 严重问题，需要改进

## 团队规模适配

### Solo 测试（1人）
```markdown
重点:
- 优先核心功能测试
- 最大化自动化覆盖
- 简化流程和文档
- 使用工具提高效率

时间分配:
- 功能测试: 50%
- 自动化维护: 30%
- 缺陷管理: 15%
- 文档和沟通: 5%
```

### 小团队（2-3人）
```markdown
分工:
- 测试负责人: 计划、协调、关键测试
- 测试工程师: 功能测试、自动化
- (可选) 专项测试: 性能、安全等

协作:
- 每日站会: 15分钟
- 结对测试: 复杂场景
- 知识分享: 每周一次
```

### 大团队（4+人）
```markdown
分工:
- 测试经理: 策略、计划、协调
- 功能测试: 2-3人
- 自动化测试: 1-2人
- 专项测试: 1人（性能/安全）

协作:
- 每日站会: 15分钟
- 周会: 1小时
- 月度回顾: 2小时
```

- 测试计划

## 常见问题

**Q: 测试任务太多，时间不够怎么办？**  
A: 基于风险优先级排序，先测试核心功能和高风险区域，考虑增加自动化。

**Q: 如何平衡新功能测试和回归测试？**  
A: 新功能优先，回归测试自动化，根据变更影响选择回归范围。

**Q: 发现的问题太多，如何处理？**  
A: 按严重程度分类，阻塞问题立即处理，一般问题记录跟踪。

**Q: 自动化测试经常失败怎么办？**  
A: 分析失败原因，修复 flaky 测试，优化测试稳定性。

## 工具推荐

### 测试管理
- **Jira**: 任务和缺陷管理
- **TestRail**: 测试用例管理
- **Zephyr**: Jira 测试插件

### 自动化测试
- **Playwright**: Web 自动化
- **Selenium**: 跨浏览器测试
- **Pytest**: Python 测试框架

### 协作工具
- **Slack/Teams**: 团队沟通
- **Confluence**: 文档协作
- **Miro**: 可视化协作

## 最佳实践

1. **早测试，常测试**: 尽早介入，持续测试
2. **自动化优先**: 重复性工作自动化
3. **风险驱动**: 基于风险确定测试重点
4. **快速反馈**: 及时发现和报告问题
5. **持续改进**: 定期回顾和优化流程
