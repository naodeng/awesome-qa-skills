# Bug 修复验证工作流

## 工作流概述

**适用场景**: 验证 Bug 修复的完整流程  
**涉及 Skills**: bug-reporting, functional-testing, test-case-reviewer  
**预计时间**: 0.5-2 天

---

## 工作流步骤

### 1. Bug 分析阶段

**使用 Skill**: `bug-reporting`

**输入**:
- Bug 报告
- 修复代码变更

**活动**:
```markdown
1. 理解 Bug 的根本原因
2. 确认修复方案的合理性
3. 识别可能的副作用
4. 确定验证范围
```

**输出**:
- Bug 验证计划
- 测试范围清单

---

### 2. 验证测试设计

**使用 Skill**: `test-case-writing`

**输入**:
- Bug 验证计划
- 原始 Bug 复现步骤

**活动**:
```markdown
1. 设计 Bug 复现测试用例
2. 设计相关场景测试用例
3. 设计回归测试用例
4. 准备测试数据
```

**输出**:
- Bug 验证测试用例
- 回归测试用例

---

### 3. Bug 验证执行

**使用 Skill**: `functional-testing`

**输入**:
- Bug 验证测试用例
- 修复后的版本

**活动**:
```markdown
1. 验证 Bug 已修复（原问题不再出现）
2. 验证修复没有引入新问题
3. 执行相关功能的回归测试
4. 验证不同环境和配置
```

**输出**:
- 验证测试报告
- Bug 状态更新

---

### 4. 测试用例评审

**使用 Skill**: `test-case-reviewer`

**输入**:
- Bug 验证测试用例
- 测试执行结果

**活动**:
```markdown
1. 评审测试用例的完整性
2. 确认测试覆盖是否充分
3. 将有价值的用例加入回归套件
4. 更新测试文档
```

**输出**:
- 更新的测试套件
- 测试用例库

## Bug 验证决策树

```
Bug 修复完成
    ↓
能否复现原 Bug？
    ├─ 是 → Bug 未修复，返回开发
    └─ 否 → 继续
         ↓
相关功能是否正常？
    ├─ 否 → 发现新问题，记录缺陷
    └─ 是 → 继续
         ↓
回归测试是否通过？
    ├─ 否 → 发现回归问题，记录缺陷
    └─ 是 → Bug 验证通过
```

---

## 最佳实践

1. **理解根因**: 不仅验证表面问题，要理解根本原因
2. **扩展测试**: 测试相似场景和边界条件
3. **回归保护**: 将 Bug 用例加入回归测试套件
4. **文档记录**: 记录 Bug 原因和验证方法
5. **知识分享**: 分享典型 Bug 和测试经验

---

## 常见问题

**Q: Bug 无法复现怎么办？**  
A: 检查环境配置、数据状态、操作步骤，与报告人沟通确认。

**Q: 修复后发现新问题怎么办？**  
A: 记录新缺陷，评估严重程度，决定是否阻塞原 Bug 关闭。

**Q: 如何确定回归测试范围？**  
A: 基于代码变更影响分析，测试相关模块和依赖功能。

**Q: Bug 验证需要多长时间？**  
A: 取决于 Bug 复杂度，简单 Bug 可能几分钟，复杂 Bug 可能需要几天。
