# 新功能开发测试工作流

## 工作流概述

**适用场景**: 开发新功能时的完整测试流程  
**涉及 Skills**: requirements-analysis, test-case-writing, functional-testing, automation-testing  
**预计时间**: 2-5 天

---

## 工作流步骤

### 1. 需求分析阶段

**使用 Skill**: `requirements-analysis`

**输入**:
- 产品需求文档（PRD）
- 用户故事
- 验收标准

**活动**:
```markdown
1. 使用 5W1H 方法分析需求
2. 识别测试点和边界条件
3. 确定测试范围和优先级
4. 识别潜在风险
```

**输出**:
- 需求分析报告
- 测试点清单
- 风险评估

---

### 2. 测试设计阶段

**使用 Skill**: `test-case-writing`

**输入**:
- 需求分析报告
- 测试点清单

**活动**:
```markdown
1. 应用等价类划分设计测试用例
2. 应用边界值分析设计边界测试
3. 设计正向和负向测试场景
4. 评审测试用例
```

**输出**:
- 测试用例集
- 测试数据准备清单

---

### 3. 功能测试阶段

**使用 Skill**: `functional-testing`

**输入**:
- 测试用例集
- 测试环境

**活动**:
```markdown
1. 准备测试环境和测试数据
2. 执行功能测试用例
3. 记录测试结果和缺陷
4. 回归测试修复的缺陷
```

**输出**:
- 测试执行报告
- 缺陷报告

---

### 4. 自动化测试阶段

**使用 Skill**: `automation-testing`

**输入**:
- 稳定的功能
- 核心测试用例

**活动**:
```markdown
1. 识别适合自动化的测试用例
2. 编写自动化测试脚本
3. 集成到 CI/CD 流程
4. 维护自动化测试套件
```

**输出**:
- 自动化测试脚本
- 自动化测试报告

---

## 检查清单

### 需求分析阶段
- [ ] 需求已完全理解
- [ ] 测试点已全部识别
- [ ] 边界条件已明确
- [ ] 风险已评估

### 测试设计阶段
- [ ] 测试用例覆盖所有需求
- [ ] 包含正向和负向场景
- [ ] 测试用例已评审
- [ ] 测试数据已准备

### 功能测试阶段
- [ ] 测试环境已就绪
- [ ] 所有测试用例已执行
- [ ] 缺陷已记录和跟踪
- [ ] 回归测试已完成

### 自动化测试阶段
- [ ] 核心用例已自动化
- [ ] 自动化脚本已评审
- [ ] 已集成到 CI/CD
- [ ] 自动化测试通过

---

## 最佳实践

1. **尽早介入**: 在需求阶段就开始测试分析
2. **持续沟通**: 与开发团队保持密切沟通
3. **优先级管理**: 先测试核心功能和高风险区域
4. **自动化优先**: 对稳定的核心功能尽早自动化
5. **文档记录**: 保持完整的测试文档和缺陷记录

---

## 常见问题

**Q: 什么时候开始编写测试用例？**  
A: 在需求明确后立即开始，不要等到开发完成。

**Q: 如何确定自动化的优先级？**  
A: 优先自动化核心业务流程、高频执行的用例和回归测试。

**Q: 如何处理需求变更？**  
A: 及时更新测试用例，评估变更影响，调整测试计划。
