# Sprint 测试工作流（迭代测试）

## 工作流概述

**适用场景**: 敏捷开发中的 Sprint 测试流程  
**涉及 Skills**: test-strategy, test-case-writing, functional-testing, automation-testing, test-reporting  
**Sprint 周期**: 1-4 周

---

## Sprint 测试时间线

```
Sprint 开始
    ↓
Sprint 规划（Day 1）
    ↓
开发和测试（Day 2-N-2）
    ↓
回归测试（Day N-1）
    ↓
Sprint 评审和回顾（Day N）
    ↓
Sprint 结束
```

---

## 阶段1: Sprint 规划（Day 1）

### 1.1 参与 Sprint 规划会议

**使用 Skill**: `test-strategy`

**活动**:
```markdown
1. 理解 Sprint 目标和用户故事
2. 参与故事点估算
3. 识别测试风险和依赖
4. 确定验收标准
```

**输出**:
- Sprint 测试目标
- 测试风险清单
- 资源需求

### 1.2 制定 Sprint 测试计划

**Sprint 规划检查清单**:
```markdown
## 需求理解
- [ ] 所有用户故事已理解
- [ ] 验收标准已明确
- [ ] 技术实现方案已了解
- [ ] 依赖关系已识别

## 测试准备
- [ ] 测试环境已确认
- [ ] 测试数据需求已明确
- [ ] 测试工具已准备
- [ ] 自动化策略已确定

## 风险评估
- [ ] 技术风险已识别
- [ ] 时间风险已评估
- [ ] 资源风险已考虑
- [ ] 缓解措施已制定

## 团队协作
- [ ] 与开发团队对齐
- [ ] 与产品团队对齐
- [ ] 测试任务已分配
- [ ] 沟通机制已建立
```

**输出**:
- Sprint 测试计划
- 测试任务分配
- 测试时间表

---

## 阶段2: 开发和测试（Day 2 到 Day N-2）

### 2.1 测试用例设计

**使用 Skill**: `test-case-writing`

**活动**:
```markdown
1. 基于用户故事编写测试用例
2. 设计正向和负向场景
3. 设计边界和异常测试
4. 评审测试用例
```

**测试用例覆盖**:
- ✅ 功能需求
- ✅ 验收标准
- ✅ 边界条件
- ✅ 异常场景
- ✅ 集成点

### 2.2 持续测试

**使用 Skill**: `functional-testing`

**每日活动**:
```markdown
1. 参加每日站会
2. 获取最新构建
3. 执行测试用例
4. 记录和跟踪缺陷
5. 与开发团队协作
```

**测试节奏**:
```
每日:
- 冒烟测试（每次构建）
- 功能测试（新完成的故事）
- 缺陷验证（已修复的缺陷）

每周:
- 集成测试
- 探索性测试
- 自动化测试维护
```

### 2.3 缺陷管理

**缺陷优先级**:
| 优先级 | 定义 | 处理时间 |
|--------|------|----------|
| P0 | 阻塞性缺陷 | 立即修复 |
| P1 | 严重缺陷 | 当前 Sprint |
| P2 | 一般缺陷 | 当前或下个 Sprint |
| P3 | 轻微缺陷 | Backlog |

**缺陷跟踪**:
```markdown
- [ ] 每日更新缺陷状态
- [ ] 验证已修复的缺陷
- [ ] 跟踪阻塞性问题
- [ ] 与团队同步缺陷情况
```

### 2.4 自动化测试

**使用 Skill**: `automation-testing`

**活动**:
```markdown
1. 识别适合自动化的用例
2. 编写自动化测试脚本
3. 集成到 CI/CD
4. 维护自动化测试套件
```

**自动化策略**:
- 核心业务流程优先
- 回归测试用例优先
- 稳定功能优先
- 高频执行用例优先

---

## 阶段3: 回归测试（Day N-1）

### 3.1 回归测试执行

**回归测试优先级矩阵**:

| 变更影响 | 业务重要性 | 优先级 | 测试策略 |
|----------|-----------|--------|----------|
| 高 | 高 | P0 | 完整测试 |
| 高 | 中 | P1 | 重点测试 |
| 高 | 低 | P2 | 基本测试 |
| 中 | 高 | P1 | 重点测试 |
| 中 | 中 | P2 | 基本测试 |
| 中 | 低 | P3 | 抽样测试 |
| 低 | 高 | P2 | 基本测试 |
| 低 | 中 | P3 | 抽样测试 |
| 低 | 低 | P4 | 可选测试 |

**回归测试检查清单**:
```markdown
- [ ] 自动化回归测试已执行
- [ ] 核心功能手工验证已完成
- [ ] 集成测试已通过
- [ ] 性能测试已完成（如需要）
- [ ] 安全测试已完成（如需要）
- [ ] 所有 P0/P1 缺陷已修复
- [ ] 回归测试报告已生成
```

### 3.2 发布准备

**发布检查清单**:
```markdown
## 测试完成度
- [ ] 所有测试用例已执行
- [ ] 测试覆盖率达标（>80%）
- [ ] 自动化测试通过率 >95%
- [ ] 无阻塞性缺陷

## 质量指标
- [ ] 缺陷密度可接受
- [ ] 性能指标达标
- [ ] 安全扫描通过
- [ ] 用户体验良好

## 文档和交付
- [ ] 测试报告已完成
- [ ] 已知问题已记录
- [ ] 发布说明已准备
- [ ] 回滚计划已制定
```

## 阶段4: Sprint 评审和回顾（Day N）

### 4.1 Sprint 评审

**使用 Skill**: `test-reporting`

**活动**:
```markdown
1. 演示测试成果
2. 展示测试覆盖情况
3. 说明质量状况
4. 讨论已知问题
```

**Sprint 测试总结模板**:
```markdown

## Sprint 概况
- Sprint 目标: [目标]
- 完成故事: [数量]
- 测试周期: [日期范围]

## 测试执行
- 计划用例: [数量]
- 执行用例: [数量]
- 通过用例: [数量]
- 失败用例: [数量]
- 通过率: [百分比]

## 缺陷统计
- 新增缺陷: [数量]
  - P0: [数量]
  - P1: [数量]
  - P2: [数量]
  - P3: [数量]
- 已修复: [数量]
- 待修复: [数量]

## 自动化进展
- 新增自动化用例: [数量]
- 自动化覆盖率: [百分比]
- 自动化通过率: [百分比]

## 质量评估
- 整体质量: [优秀/良好/一般/较差]
- 发布建议: [可发布/有条件发布/不建议发布]
- 主要风险: [风险描述]
```

### 4.2 Sprint 回顾

**回顾议题**:
```markdown
## 做得好的
- [亮点 1]
- [亮点 2]
- [亮点 3]

## 需要改进的
- [问题 1] → [改进措施]
- [问题 2] → [改进措施]
- [问题 3] → [改进措施]

## 行动项
- [ ] [行动项 1] - 负责人: [姓名]
- [ ] [行动项 2] - 负责人: [姓名]
- [ ] [行动项 3] - 负责人: [姓名]
```

**度量指标回顾**:
| 指标 | 目标 | 实际 | 趋势 |
|------|------|------|------|
| 测试覆盖率 | 80% | 85% | ↑ |

| 缺陷发现率 | - | 15 | → |
| 缺陷修复率 | 90% | 92% | ↑ |
| 测试效率 | - | 良好 | → |

## 团队规模适配

### 小团队（1-2人）
```markdown
策略:
- 重点测试核心功能
- 最大化自动化
- 简化文档和流程
- 与开发紧密协作

时间分配:
- 测试执行: 60%
- 自动化: 25%
- 计划和报告: 15%
```

### 中等团队（3-5人）
```markdown
分工:
- 测试负责人: 计划、协调、关键测试
- 功能测试: 2-3人
- 自动化测试: 1人

协作:
- 每日站会
- 周中同步
- Sprint 回顾
```

### 大团队（6+人）
```markdown
分工:
- 测试经理: 策略和管理
- 功能测试: 3-4人
- 自动化测试: 2人
- 专项测试: 1人

协作:
- 每日站会
- 周会
- Sprint 规划和回顾
- 跨团队协作
```

## 常见问题

**Q: Sprint 中途需求变更怎么办？**  
A: 评估影响，调整测试计划，与团队沟通，必要时调整 Sprint 范围。

**Q: 测试时间不够怎么办？**  
A: 基于风险优先级测试，增加自动化，考虑延长 Sprint 或减少范围。

**Q: 开发延期影响测试怎么办？**  
A: 及早发现和沟通，调整测试策略，必要时延期发布。

**Q: 如何平衡新功能测试和技术债务？**  
A: 在 Sprint 规划时预留时间，逐步偿还技术债务。

---

## 最佳实践

1. **左移测试**: 尽早介入，参与需求评审
2. **持续测试**: 每日测试，快速反馈
3. **自动化优先**: 重复性工作自动化
4. **团队协作**: 与开发紧密合作
5. **持续改进**: 每个 Sprint 回顾和优化
