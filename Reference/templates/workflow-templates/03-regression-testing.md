# 回归测试工作流

## 工作流概述

**适用场景**: 版本发布前的回归测试  
**涉及 Skills**: test-strategy, automation-testing, test-reporting  
**预计时间**: 1-3 天

---

## 工作流步骤

### 1. 回归测试计划

**使用 Skill**: `test-strategy`

**输入**:
- 代码变更清单
- 历史缺陷数据
- 测试用例库

**活动**:
```markdown
1. 分析代码变更影响范围
2. 确定回归测试范围和优先级
3. 选择回归测试用例
4. 制定测试计划和时间表
```

**输出**:
- 回归测试计划
- 测试用例清单
- 测试时间表

---

### 2. 测试环境准备

**输入**:
- 回归测试计划
- 测试环境要求

**活动**:
```markdown
1. 准备测试环境
2. 部署待测试版本
3. 准备测试数据
4. 验证环境就绪
```

**输出**:
- 就绪的测试环境
- 测试数据集

---

### 3. 回归测试执行

**使用 Skill**: `automation-testing`, `functional-testing`

**输入**:
- 回归测试用例
- 测试环境

**活动**:
```markdown
1. 执行自动化回归测试
2. 执行手工回归测试
3. 记录测试结果和缺陷
4. 分析失败用例
```

**输出**:
- 测试执行结果
- 缺陷报告
- 失败用例分析

---

### 4. 测试报告和决策

**使用 Skill**: `test-reporting`

**输入**:
- 测试执行结果
- 缺陷统计

**活动**:
```markdown
1. 生成回归测试报告
2. 分析测试覆盖率和通过率
3. 评估发布风险
4. 提供发布建议
```

**输出**:
- 回归测试报告
- 发布风险评估
- Go/No-Go 建议

## 回归测试策略

### 1. 完全回归
**适用**: 重大版本发布、架构变更  
**范围**: 执行所有测试用例  
**时间**: 3-5 天

### 2. 选择性回归
**适用**: 常规版本发布  
**范围**: 基于风险和变更影响选择用例  
**时间**: 1-2 天

### 3. 渐进式回归
**适用**: 持续集成环境  
**范围**: 每次提交执行核心用例，定期执行完整回归  
**时间**: 持续进行

---

## 自动化回归测试最佳实践

1. **分层测试**: 单元测试 → 集成测试 → UI 测试
2. **快速反馈**: 核心用例在 15 分钟内完成
3. **稳定可靠**: 减少 flaky 测试，提高可信度
4. **持续维护**: 及时更新失效的自动化用例
5. **并行执行**: 利用并行化缩短执行时间

## 常见问题

**Q: 如何确定回归测试范围？**  
A: 基于代码变更影响分析、历史缺陷热点、业务重要性综合确定。

**Q: 回归测试发现新缺陷怎么办？**  
A: 评估缺陷严重程度，决定是否阻塞发布，修复后重新回归。

**Q: 时间不够如何做回归测试？**  
A: 采用基于风险的测试策略，优先测试核心功能和高风险区域。

**Q: 如何提高回归测试效率？**  
A: 提高自动化覆盖率、优化测试用例、并行执行、持续集成。
