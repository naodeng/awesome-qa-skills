# 性能测试工作流 | Performance Testing Workflow

**版本**: 1.0.0  
**最后更新**: 2026-02-09  
**适用场景**: 系统性能验证、容量规划、性能优化

---

## 工作流概述 | Workflow Overview

性能测试工作流用于评估系统在不同负载条件下的表现，包括响应时间、吞吐量、资源使用率等关键指标。

### 适用场景
- 新版本发布前的性能验证
- 系统容量规划
- 性能瓶颈定位
- 性能优化效果验证
- SLA 合规性检查

### 涉及的 Skills
- `test-strategy` - 性能测试策略制定
- `performance-testing` - 性能测试执行
- `test-reporting` - 性能测试报告

---

## 工作流步骤 | Workflow Steps

### 阶段 1: 性能测试规划 (Planning)

#### 1.1 确定性能目标
- [ ] 定义性能指标（响应时间、TPS、并发数等）
- [ ] 设定性能基线和目标值
- [ ] 确定 SLA 要求
- [ ] 识别关键业务场景

**输入**:
- 系统架构文档
- 业务需求文档
- SLA 协议

**输出**:
- 性能测试目标文档
- 性能指标定义

**使用 Skill**: `test-strategy`

**提示词示例**:
```
请根据以下系统信息制定性能测试策略：

系统类型：[Web API / 移动应用 / 微服务]
预期用户量：[日活/月活]
关键业务场景：[列出 3-5 个]
SLA 要求：[响应时间、可用性等]

请输出：
1. 性能测试目标
2. 关键性能指标（KPI）
3. 测试场景优先级
4. 资源需求评估
```

#### 1.2 设计测试场景
- [ ] 识别典型用户行为模式
- [ ] 设计负载模型（正常、峰值、压力）
- [ ] 确定测试数据需求
- [ ] 规划测试环境

**测试类型**:
- **负载测试 (Load Testing)**: 验证系统在预期负载下的表现
- **压力测试 (Stress Testing)**: 找出系统的极限承载能力
- **峰值测试 (Spike Testing)**: 验证系统应对突发流量的能力
- **稳定性测试 (Endurance Testing)**: 验证系统长时间运行的稳定性
- **容量测试 (Volume Testing)**: 验证系统处理大数据量的能力

#### 1.3 准备测试环境
- [ ] 搭建性能测试环境
- [ ] 配置监控工具
- [ ] 准备测试数据
- [ ] 验证环境配置

**环境要求**:
- 与生产环境配置一致或相似
- 独立的测试环境，避免干扰
- 完整的监控体系（APM、日志、指标）

---

### 阶段 2: 测试脚本开发 (Script Development)

#### 2.1 编写性能测试脚本
- [ ] 选择性能测试工具（K6、JMeter、Gatling、Locust）
- [ ] 实现测试场景脚本
- [ ] 参数化测试数据
- [ ] 添加检查点和断言

**使用 Skill**: `performance-testing`

**提示词示例**:
```
请为以下 API 编写 K6 性能测试脚本：

API 端点：POST /api/orders
请求体：
{
  "userId": "{{userId}}",
  "items": [
    {"productId": "{{productId}}", "quantity": {{quantity}}}
  ]
}

测试场景：
- 虚拟用户数：100
- 持续时间：5 分钟
- 目标 TPS：500
- 响应时间要求：< 200ms (P95)

请包含：
1. 完整的 K6 脚本
2. 数据参数化
3. 性能阈值设置
4. 自定义指标收集
```

#### 2.2 脚本调试和验证
- [ ] 单用户场景验证
- [ ] 小规模负载测试
- [ ] 检查监控数据采集
- [ ] 优化脚本性能

**验证清单**:
- ✓ 脚本能正常执行
- ✓ 请求成功率 > 99%
- ✓ 监控数据正常采集
- ✓ 测试数据正确使用

---

### 阶段 3: 测试执行 (Execution)

#### 3.1 基线测试
- [ ] 执行单用户测试
- [ ] 记录基线性能数据
- [ ] 验证系统功能正常

**目的**: 建立性能基线，作为后续对比的参考

#### 3.2 负载测试
- [ ] 逐步增加负载
- [ ] 监控系统资源使用
- [ ] 记录性能指标
- [ ] 识别性能拐点

**负载递增策略**:
```
阶段 1: 10 用户 - 5 分钟
阶段 2: 50 用户 - 10 分钟
阶段 3: 100 用户 - 15 分钟
阶段 4: 200 用户 - 15 分钟
阶段 5: 500 用户 - 10 分钟
```

#### 3.3 压力测试
- [ ] 持续增加负载直到系统崩溃
- [ ] 记录系统极限值
- [ ] 观察系统恢复能力
- [ ] 分析失败原因

#### 3.4 峰值测试
- [ ] 模拟突发流量
- [ ] 验证自动扩展机制
- [ ] 检查系统稳定性
- [ ] 评估恢复时间

#### 3.5 稳定性测试
- [ ] 长时间运行测试（4-24 小时）
- [ ] 监控内存泄漏
- [ ] 检查资源释放
- [ ] 验证系统稳定性

**监控指标**:
- CPU 使用率
- 内存使用率
- 磁盘 I/O
- 网络带宽
- 数据库连接数
- 线程池状态
- GC 频率和时长

---

### 阶段 4: 结果分析 (Analysis)

#### 4.1 数据收集和整理
- [ ] 汇总测试结果
- [ ] 整理监控数据
- [ ] 生成性能图表
- [ ] 对比历史数据

**关键指标**:
- **响应时间**: 平均值、P50、P95、P99、最大值
- **吞吐量**: TPS/QPS
- **错误率**: 失败请求百分比
- **并发数**: 同时在线用户数
- **资源使用**: CPU、内存、网络、磁盘

#### 4.2 性能瓶颈分析
- [ ] 识别性能瓶颈
- [ ] 分析根本原因
- [ ] 提出优化建议
- [ ] 评估优化优先级

**常见瓶颈**:
- 数据库查询慢
- 缓存命中率低
- 网络延迟高
- 代码逻辑复杂
- 资源配置不足
- 第三方服务慢

#### 4.3 生成测试报告
- [ ] 编写性能测试报告
- [ ] 包含图表和数据
- [ ] 提供优化建议
- [ ] 评估风险等级

**使用 Skill**: `test-reporting`

**提示词示例**:
```
请根据以下性能测试结果生成测试报告：

测试场景：订单创建 API 负载测试
测试时间：2026-02-09 10:00-11:00
虚拟用户数：100 → 500（递增）

测试结果：
- 平均响应时间：150ms
- P95 响应时间：280ms
- P99 响应时间：450ms
- 最大 TPS：480
- 错误率：0.5%
- CPU 使用率：65%（峰值 85%）
- 内存使用率：70%（峰值 82%）

发现的问题：
1. 数据库查询慢（平均 80ms）
2. 缓存命中率低（60%）
3. 500 用户时错误率上升到 2%

请生成包含以下内容的报告：
1. 执行摘要
2. 测试环境和配置
3. 测试结果详细分析
4. 性能瓶颈和根因分析
5. 优化建议
6. 风险评估
```

---

### 阶段 5: 优化和验证 (Optimization)

#### 5.1 性能优化
- [ ] 实施优化方案
- [ ] 代码优化
- [ ] 配置调优
- [ ] 架构改进

**优化方向**:
- 数据库优化（索引、查询优化、连接池）
- 缓存策略（Redis、CDN、本地缓存）
- 代码优化（算法、并发、异步）
- 架构优化（负载均衡、微服务、消息队列）
- 资源扩容（垂直扩展、水平扩展）

#### 5.2 回归测试
- [ ] 重新执行性能测试
- [ ] 对比优化前后数据
- [ ] 验证优化效果
- [ ] 确认无功能回归

#### 5.3 持续监控
- [ ] 部署到生产环境
- [ ] 配置性能监控告警
- [ ] 定期性能回归测试
- [ ] 建立性能基线库

## 性能测试工具选择 | Tool Selection

### K6
- **优点**: 现代化、脚本简单、云原生
- **适用**: API 测试、微服务测试
- **示例**: `skills/testing-types/performance-testing/examples/k6-load-test`

### JMeter
- **优点**: 功能强大、插件丰富、社区活跃
- **适用**: Web 应用、API、数据库测试
- **学习曲线**: 中等

### Gatling
- **优点**: 高性能、Scala DSL、实时报告
- **适用**: 高并发场景、复杂业务流程
- **学习曲线**: 较高

### Locust
- **优点**: Python 编写、易于扩展、分布式
- **适用**: Python 项目、自定义场景
- **学习曲线**: 低

## 性能指标参考 | Performance Metrics Reference

### 响应时间标准
| 场景 | 优秀 | 良好 | 可接受 | 需优化 |
|------|------|------|--------|--------|
| API 查询 | < 100ms | < 200ms | < 500ms | > 500ms |
| API 写入 | < 200ms | < 500ms | < 1s | > 1s |
| 页面加载 | < 1s | < 2s | < 3s | > 3s |
| 复杂计算 | < 2s | < 5s | < 10s | > 10s |

### 吞吐量标准
| 系统规模 | 小型 | 中型 | 大型 | 超大型 |
|----------|------|------|------|--------|
| TPS | < 100 | 100-1000 | 1000-10000 | > 10000 |
| 并发用户 | < 100 | 100-1000 | 1000-10000 | > 10000 |

### 资源使用标准
| 资源 | 正常 | 警告 | 危险 |
|------|------|------|------|
| CPU | < 70% | 70-85% | > 85% |
| 内存 | < 70% | 70-85% | > 85% |
| 磁盘 I/O | < 70% | 70-85% | > 85% |
| 网络带宽 | < 70% | 70-85% | > 85% |

## 常见问题和解决方案 | Troubleshooting

### 问题 1: 响应时间过长
**可能原因**:
- 数据库查询慢
- 网络延迟
- 代码逻辑复杂
- 资源不足

**解决方案**:
- 优化数据库查询和索引
- 使用缓存减少数据库访问
- 优化代码逻辑
- 增加服务器资源

### 问题 2: 吞吐量不足
**可能原因**:
- 并发处理能力不足
- 资源瓶颈
- 锁竞争
- 第三方服务限流

**解决方案**:
- 增加线程池大小
- 使用异步处理
- 优化锁策略
- 实现请求队列

### 问题 3: 错误率高
**可能原因**:
- 资源耗尽
- 超时设置不合理
- 并发冲突
- 数据库连接池不足

**解决方案**:
- 增加资源配置
- 调整超时时间
- 优化并发控制
- 增加连接池大小

### 问题 4: 内存泄漏
**可能原因**:
- 对象未释放
- 缓存无限增长
- 连接未关闭
- 循环引用

**解决方案**:
- 使用内存分析工具
- 实现缓存淘汰策略
- 确保资源正确关闭
- 优化对象生命周期

## 最佳实践 | Best Practices

### 测试设计
1. **从小到大**: 先小规模测试，再逐步增加负载
2. **真实场景**: 模拟真实用户行为和数据分布
3. **隔离环境**: 使用独立的测试环境
4. **完整监控**: 覆盖应用、数据库、中间件、系统资源

### 测试执行
1. **预热系统**: 正式测试前进行预热
2. **稳定观察**: 每个负载阶段保持足够时间
3. **记录详细**: 记录所有配置和环境信息
4. **重复验证**: 多次执行确保结果稳定

### 结果分析
1. **多维度分析**: 结合响应时间、吞吐量、资源使用
2. **对比分析**: 与历史数据、基线、目标对比
3. **根因分析**: 深入分析性能瓶颈的根本原因
4. **量化评估**: 用数据说话，避免主观判断

### 持续改进
1. **建立基线**: 每次发布建立性能基线
2. **自动化测试**: 集成到 CI/CD 流程
3. **定期回归**: 定期执行性能回归测试
4. **性能文化**: 将性能作为质量的重要指标
