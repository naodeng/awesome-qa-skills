# 移动端测试工作流 | Mobile Testing Workflow

**版本**: 1.0.0  
**最后更新**: 2026-02-09  
**适用场景**: Android/iOS 应用测试、跨平台应用测试

---

## 工作流概述

移动端测试工作流覆盖移动应用的功能、兼容性、性能、网络、安全等全方位测试。

### 适用场景
- 原生应用测试（Android/iOS）
- 跨平台应用测试（React Native/Flutter）
- 混合应用测试（Hybrid）
- 移动 Web 测试

### 涉及的 Skills
- `requirements-analysis` - 需求分析
- `test-strategy` - 测试策略
- `mobile-testing` - 移动端测试
- `automation-testing` - 自动化测试
- `performance-testing` - 性能测试
- `test-reporting` - 测试报告

## 工作流步骤

### 阶段 1: 测试规划

#### 1.1 需求分析
- [ ] 分析功能需求
- [ ] 识别目标平台（Android/iOS）
- [ ] 确定支持的系统版本
- [ ] 定义设备矩阵

**使用 Skill**: `requirements-analysis`

**提示词示例**:
```
请分析以下移动应用的测试需求：

应用类型：[电商 / 社交 / 工具类]
平台：[Android / iOS / 跨平台]
目标用户：[描述用户群体]
核心功能：[列出 5-10 个核心功能]

请输出：
1. 功能测试范围
2. 兼容性测试矩阵
3. 性能测试指标
4. 特殊测试场景（网络、权限、生命周期等）
```

#### 1.2 制定测试策略
- [ ] 确定测试类型和优先级
- [ ] 规划自动化测试范围
- [ ] 选择测试工具和框架
- [ ] 评估资源需求

**使用 Skill**: `test-strategy`

**测试类型**:
- 功能测试
- 兼容性测试（设备、系统版本）
- 性能测试（启动时间、内存、电量）
- 网络测试（弱网、断网、切换）
- 安全测试（数据加密、权限）
- 用户体验测试

---

### 阶段 2: 测试环境准备

#### 2.1 设备准备
- [ ] 准备真机测试设备
- [ ] 配置模拟器/仿真器
- [ ] 安装测试工具
- [ ] 配置测试账号

**设备矩阵示例**:
| 平台 | 系统版本 | 设备型号 | 屏幕尺寸 |
|------|----------|----------|----------|
| Android | 11, 12, 13, 14 | Pixel, Samsung, Xiaomi | 5.5", 6.1", 6.7" |
| iOS | 15, 16, 17 | iPhone 12, 13, 14, 15 | 5.4", 6.1", 6.7" |

#### 2.2 自动化环境搭建
- [ ] 安装 Appium / Detox / Espresso / XCUITest
- [ ] 配置测试框架
- [ ] 编写测试脚本
- [ ] 验证环境可用性

---

### 阶段 3: 功能测试

#### 3.1 核心功能测试
- [ ] 用户注册和登录
- [ ] 主要业务流程
- [ ] 数据输入和验证
- [ ] 页面导航和跳转

**使用 Skill**: `mobile-testing`

**提示词示例**:
```
请为以下移动应用功能设计测试用例：

功能：用户登录
平台：Android + iOS
登录方式：手机号 + 验证码、第三方登录（微信、Apple）

请覆盖：
1. 正常登录流程
2. 异常场景（网络异常、验证码错误、账号被封禁）
3. 边界条件（手机号格式、验证码过期）
4. 兼容性（不同系统版本、不同设备）
5. 安全性（密码加密、Token 管理）

请输出详细的测试用例。
```

#### 3.2 UI/UX 测试
- [ ] 界面布局和适配
- [ ] 交互体验（触摸、滑动、手势）
- [ ] 动画和过渡效果
- [ ] 多语言和国际化

---

### 阶段 4: 兼容性测试

#### 4.1 设备兼容性
- [ ] 不同品牌设备测试
- [ ] 不同屏幕尺寸适配
- [ ] 不同分辨率测试
- [ ] 横竖屏切换测试

#### 4.2 系统版本兼容性
- [ ] 主流系统版本测试
- [ ] 新旧版本兼容性
- [ ] 系统特性适配（暗黑模式、分屏等）

---

### 阶段 5: 专项测试

#### 5.1 性能测试
- [ ] 启动时间测试
- [ ] 内存使用测试
- [ ] CPU 使用测试
- [ ] 电量消耗测试
- [ ] 流量消耗测试

**使用 Skill**: `performance-testing`

**性能指标**:
- 冷启动时间 < 3s
- 热启动时间 < 1s
- 内存占用 < 200MB
- CPU 使用率 < 30%（空闲时）
- 电量消耗 < 5%/小时

#### 5.2 网络测试
- [ ] 弱网环境测试（2G/3G/4G/5G）
- [ ] 断网重连测试
- [ ] 网络切换测试（WiFi ↔ 移动网络）
- [ ] 超时和重试机制测试

**网络场景**:
- 正常网络
- 弱网（延迟 500ms+）
- 断网
- 网络切换
- 丢包（10%+）

#### 5.3 生命周期测试
- [ ] 前后台切换
- [ ] 应用被杀死和恢复
- [ ] 来电/短信中断
- [ ] 系统通知中断
- [ ] 低电量模式

#### 5.4 权限测试
- [ ] 相机权限
- [ ] 相册权限
- [ ] 位置权限
- [ ] 通知权限
- [ ] 存储权限
- [ ] 权限拒绝场景

---

### 阶段 6: 自动化测试

#### 6.1 自动化脚本开发
- [ ] 编写 UI 自动化脚本

- [ ] 定期回归测试

## 常见问题

### 问题 1: 设备碎片化
**解决方案**: 
- 使用云测试平台（BrowserStack、Sauce Labs）
- 优先测试主流设备和系统版本
- 建立设备矩阵，覆盖关键组合

### 问题 2: 自动化测试不稳定
**解决方案**:
- 使用显式等待而非固定延迟
- 优化元素定位策略
- 处理网络延迟和异步操作
- 添加重试机制

### 问题 3: 性能问题难定位
**解决方案**:
- 使用性能分析工具（Android Profiler、Instruments）
- 监控关键性能指标
- 对比不同版本的性能数据
- 分析用户反馈和崩溃日志

---

## 最佳实践

1. **真机优先**: 关键功能必须在真机上测试
2. **自动化回归**: 核心流程实现自动化
3. **持续监控**: 线上性能和崩溃监控
4. **用户反馈**: 重视用户反馈和应用商店评价
5. **灰度发布**: 新版本先小范围灰度测试
