# K6 Report Interpretation Guide

This guide standardizes how to interpret k6 result summaries generated by `K6_SUMMARY_EXPORT`.

## 1. Required Metrics

- `http_req_duration` p(95), p(99)
- `http_req_failed` rate
- `http_reqs` rate (throughput indicator)
- `checks` pass rate
- optional custom business metrics (success ratio, business latency trend)

## 2. Default Pass/Fail Baseline

- Latency gate:
  - p95 <= 800ms
  - p99 <= 1500ms
- Error gate:
  - `http_req_failed` < 1%
- Stability gate:
  - no sustained upward degradation in soak window
- Recovery gate:
  - spike test should recover near baseline after burst window

Use scenario-specific thresholds (stress/spike/soak) when explicitly configured in test scripts.

## 3. Recommended Output Sections

For each run, report:

1. Test metadata
- test type, target env, base URL, timestamp, commit/build ID

2. Workload profile
- stages, VUs, duration, executor type

3. Key results
- p95, p99, error rate, throughput, check pass rate

4. Verdict
- pass/fail against thresholds with failed items highlighted

5. Bottleneck hints
- likely saturation domain: app CPU, DB, cache, network, dependency timeout

## 4. Trend Comparison Rules

- Compare same script + same env + same workload model first.
- For performance regression:
  - p95 increased > 15% OR
  - error rate increased > 0.5 percentage point
  - mark as regression candidate.
- Always correlate with infra/app metrics before concluding root cause.

## 5. Suggested Team Workflow

1. Run `load` as baseline for current build.
2. Run `stress` to identify upper-limit behavior.
3. Run `spike` to validate burst recovery.
4. Run `soak` for long-run stability on release candidates.
5. Use summary tools in `scripts/tools/` for quick trend review.
