# 性能测试 Skill 介绍：拿 京东线上活动秒杀礼品 这个场景说清楚

很多测试 skill 的问题不在于能力不够，而在于介绍写得太像目录。目录能说明有这个东西，博客得说明它到底拿来干什么——以及在一个真实业务里能打出什么效果。

这篇就直接讲 性能测试。不绕。这个 skill 是干什么的、什么时候该用、边界在哪里、放进 京东线上活动秒杀礼品 这个业务里到底能跑出什么结果——都在下面。

## Skill 介绍

`performance-testing` 这个 skill 的定位很明确——聚焦秒杀高峰的吞吐、延迟、库存竞争和系统稳定性。

它对应的能力描述是：Use this skill when you need to design performance testing for load, stress, spike, endurance, or capacity objectives; triggers include 性能测试 and performance testing.

这里真正有价值的点，不是“它会测”。这句话太空了。价值在于它会把测试工作压到一个可执行的形态——要么是策略，要么是用例，要么是报告，要么是能直接追问题的结果。

## 使用场景和范围

这个 skill 适合什么场景——适合压测活动入口、库存服务、消息队列和下单服务瓶颈。

如果把范围再说细一点，它更适合下面这几类任务：

- 活动需求刚落地，需要尽快把测试思路拉直
- 活动上线时间很紧，需要优先抓高风险点
- 业务链路长，跨页面、跨服务、跨状态，不想靠拍脑袋测试
- 需要把结果沉淀成能复用的交付物，而不是一次性口头结论

它不负责所有事。它负责自己那一段——但这一段往往刚好是最容易失控的地方。

## 实战模拟：京东线上活动秒杀礼品

场景先立住——京东线上活动页做一场礼品秒杀。用户进入活动页，看到倒计时，满足资格后点击抢购，完成支付，再去领取赠品。链路看起来不复杂，真正麻烦的地方在边界——资格、库存、并发、状态同步、异常回滚，哪一块出问题都能把活动搞得很难看。

这次用 `performance-testing` skill 来模拟执行，重点就盯它该盯的部分。

### 模拟输入

- 活动时间：晚 8 点开抢，持续 10 分钟
- 礼品库存：500 份
- 资格规则：PLUS 用户且近 30 天消费满 199 元
- 关键链路：活动页 -> 资格校验 -> 秒杀下单 -> 支付 -> 礼品领取 -> 礼品中心状态回显
- 风险前提：活动流量高、库存变化快、移动端和 Web 端同时开放

### 模拟执行结果

模拟结果里，礼品资格校验接口在 5 倍峰值流量下 p95 飙到 2.8 秒，库存服务出现热点 Key 竞争。最终产物不只是“服务慢”，而是明确指出了瓶颈点。

### 最终产出长什么样

这类 skill 跑完以后，正常应该拿到的不是一句“测过了”。应该是下面这种东西：

- 明确的问题定位
- 可以复现的步骤或条件
- 对业务影响的判断
- 修复建议或者下一步测试建议

这才是可用结果。不是表演性质的结果。

## 这个 Skill 的实际价值

如果你是在做活动测试，这个 skill 的价值很直接——它帮你少漏场景，少走弯路，少在上线前靠运气。

更实际一点说，它能把“我们应该测一下”这种模糊话，压成可以执行、可以追责、可以复盘的具体动作。这件事很重要——尤其在秒杀礼品这种活动场景里，问题不是会不会出，问题是你能不能在出问题前把它挖出来。

## 结尾

`performance-testing` 不是拿来当名词收藏的。它就是一个干活的 skill——用在对的场景里，结果会很扎实。

如果你现在要测的也是活动类业务，这个 skill 值得先拉出来跑一遍。很多问题不会等到上线后才出现——它们只是一直在那儿，等人发现。
